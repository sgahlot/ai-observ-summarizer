# Dependency Version Lock Summary

This document describes the dependency version locking strategy implemented to prevent version mismatch issues between local development and CI/production environments.

## Problems Solved

### 1. Python Version Mismatch
**Issue**: Tests failed in PR CI due to Python version mismatch
- Local: Python 3.12 (due to `requires-python = ">=3.11"`)
- CI: Python 3.11
- Docker mcp_server: Python 3.12

**Solution**:
- ✅ Set `requires-python = "==3.11.*"` in `pyproject.toml`
- ✅ Added `.python-version` file with `3.11`
- ✅ Fixed `src/mcp_server/Dockerfile` to use `python-311` (was `python-312`)
- ✅ All Dockerfiles now use `python-311`

### 2. FastMCP Library Version Mismatch
**Issue**: Version drift between local and Docker builds
- Local: fastmcp 2.12.4
- Docker: fastmcp 3.0.2 (due to `fastmcp>=2.11`)
- API breakage: `get_tools()` removed in FastMCP 3.x

**Solution**:
- ✅ Locked `fastmcp==3.0.2` in all requirements files
- ✅ Updated code to use FastMCP 3.x API (`list_tools()`)

## Node.js / React UI Version Locking

The OpenShift Console Plugin and React UI use a different but equally robust version locking strategy.

### JavaScript Dependency Locking Philosophy

Unlike Python where you pin versions in requirements files, JavaScript/Node.js projects:
1. Use **semver ranges** in `package.json` (e.g., `^5.0.0`)
2. Use **lock files** to pin exact versions (`yarn.lock`)
3. The lock file is the source of truth for installed versions

**This is intentional and correct** for Node.js projects:
- `package.json` defines **compatible version ranges** (e.g., `^5.0.0` = 5.0.0 to <6.0.0)
- `yarn.lock` pins the **exact version** that satisfies the range (e.g., `5.3.1`)
- Using `--frozen-lockfile` ensures builds use the exact version from `yarn.lock`

## Dependency Update Workflow

### Python Dependencies

#### Update a Single Dependency
```bash
# Edit version in pyproject.toml, then:
uv lock
uv sync

# Update component requirements.txt files manually to match
# Then test locally before committing
```

#### Update All Dependencies
```bash
# Update all to latest compatible versions
uv lock --upgrade
uv sync

# Review changes and test thoroughly
# Update requirements.txt files to match uv.lock
```

#### After Updating pyproject.toml
```bash
# Always regenerate lock file
uv lock

# Sync local environment
uv sync

# Update requirements.txt files manually
# Build and test Docker images
make build
```

### Node.js Dependencies

#### Install from Lock File
```bash
cd openshift-plugin
yarn install --frozen-lockfile
```

#### Update a Specific Package
```bash
cd openshift-plugin
yarn upgrade <package-name>
# This updates yarn.lock with the new version
git add yarn.lock
```

#### Update All Dependencies
```bash
cd openshift-plugin
# Carefully! This updates all packages
yarn upgrade --latest

# Review changes
git diff yarn.lock

# Test thoroughly
yarn test
yarn build
```

## Monitoring for Drift

### Signs of Version Drift

1. ❌ Tests pass locally but fail in CI
2. ❌ Docker builds pull different versions than local
3. ❌ API errors like "object has no attribute"
4. ❌ Import errors in deployed containers
5. ❌ React build failures with cryptic webpack errors

### Prevention Checklist

#### Python
1. ✅ Always use exact version pins (`==`) not ranges (`>=`)
2. ✅ Keep requirements.txt files in sync with pyproject.toml
3. ✅ Use `uv lock --frozen` in CI to detect drift
4. ✅ Review Dockerfile logs for version downloads
5. ✅ Test Docker images before pushing

#### Node.js
1. ✅ Always commit `yarn.lock` to git
2. ✅ Use `yarn install --frozen-lockfile` in CI and Dockerfiles
3. ✅ Never manually edit `yarn.lock`
4. ✅ Keep Node version consistent in .nvmrc, Dockerfile, CI
5. ✅ Run `yarn install` after pulling changes
